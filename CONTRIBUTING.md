## 社区价值观

### 核心价值
1. **多元尊重**：尊重不同视角，不追求单一"正确"
2. **深度思考**：重视质量而非速度，鼓励深入探索
3. **开放透明**：决策过程公开，欢迎不同意见
4. **伦理先行**：技术发展始终以伦理为约束

### 社区文化
- **火堆旁对话**：平等、真诚、温暖的交流氛围
- **留白空间**：给思考留出空间，不急于得出结论
- **递归探索**：对问题持续深入，不满足表面答案
- **知止智慧**：知道何时停止，保持适度节奏

### 行为准则
所有贡献者必须遵守我们的[行为准则](CODE_OF_CONDUCT.md)，核心包括：
- 尊重所有参与者
- 建设性批评
- 包容多样性
- 维护安全环境

## 贡献类型

### 协议改进

#### 协议规范修改
- **范围**：修改 `spec/protocol-v0.1.json`
- **要求**：必须保持向后兼容，除非是主版本更新
- **流程**：需要提交RFC（Request for Comments）讨论

#### 新光谱类型提案
- **范围**：定义新的光谱类型（如绿色生态视角）
- **要求**：必须提供完整的认知科学依据
- **模板**：使用 `templates/new-spectrum-proposal.md`

#### 约束条件优化
- **范围**：优化多元强制、留白必需等约束
- **要求**：必须有实证研究或理论支持
- **流程**：需要社区投票通过

### 实现贡献

#### 新语言实现
- **目标语言**：JavaScript、Go、Rust、Java等
- **要求**：
  - 完整实现协议规范
  - 包含测试套件
  - 提供使用文档
- **目录结构**：`implementations/<language>/`

#### 现有实现改进
- **范围**：性能优化、bug修复、功能增强
- **要求**：
  - 不破坏现有API
  - 包含测试用例
  - 提供性能对比数据

#### 工具链开发
- **类型**：验证工具、测试工具、部署工具等
- **要求**：
  - 解决具体痛点
  - 文档完整
  - 易于集成

### 文档完善

#### 技术文档
- **类型**：API文档、架构说明、部署指南
- **标准**：清晰、准确、完整
- **格式**：Markdown，支持中文和英文

#### 用户文档
- **类型**：教程、案例研究、最佳实践
- **标准**：易懂、实用、有启发性
- **要求**：基于真实使用场景

#### 研究文档
- **类型**：白皮书、研究论文、实验报告
- **标准**：学术严谨、数据可靠、结论明确
- **要求**：遵循学术引用规范

### 社区建设

#### 用户支持
- **方式**：回答问题、解决issue、提供指导
- **要求**：耐心、专业、友好

#### 社区推广
- **方式**：撰写博客、参加会议、组织活动
- **要求**：准确传达项目理念

#### 生态建设
- **方式**：开发插件、集成其他系统、建立合作
- **要求**：符合项目价值观

#### 2. Fork仓库
1. 点击GitHub页面的"Fork"按钮
2. 克隆你的fork到本地
3. 创建特性分支

```bash
git clone https://github.com/YOUR_USERNAME/prism-interconnect.git
cd prism-interconnect
git checkout -b feature/your-feature-name
```

#### 3. 开发实现
遵循项目的技术规范和代码风格。

#### 4. 提交更改
```bash
# 添加更改
git add .

git push origin feature/your-feature-name
```

1. 访问原仓库的Pull Requests页面
2. 点击"New Pull Request"
3. 选择你的分支
4. 填写PR模板

### 第四步：参与评审

#### PR模板要求
```markdown
## 变更类型
- [ ] Bug修复
- [ ] 新功能
- [ ] 文档更新
- [ ] 其他（请说明）

## 变更描述
详细描述变更内容和原因

## 测试验证
- [ ] 单元测试通过
- [ ] 集成测试通过  
- [ ] 性能测试通过
- [ ] 文档已更新

## 相关Issue
链接相关Issue，如：Closes #123

## 检查清单
- [ ] 代码遵循项目风格
- [ ] 提交信息格式正确
- [ ] 没有引入新的警告
- [ ] 已添加测试用例
- [ ] 文档已相应更新
```

#### 评审流程
1. **自动化检查**：CI/CD流水线运行测试
2. **维护者评审**：至少1名维护者审核
3. **社区反馈**：开放给社区评论
4. **合并决策**：基于评审结果决定

#### 常见反馈类型
- **请求更改**：需要修改后重新评审
- **批准**：可以合并，但可能有建议
- **评论**：提供信息或问题，不影响合并

```python
# tests/test_prism_agent.py
import unittest

class TestPrismAgent(unittest.TestCase):
    def setUp(self):
        self.agent = PrismAgent()
    
    def test_refract_returns_valid_message(self):
        """测试折射返回有效消息"""
        result = self.agent.refract("测试困惑")
        self.assertEqual(result["protocol"], "PIP")
        self.assertGreaterEqual(len(result["spectrums"]), 3)
    
    def test_recursion_depth_increments(self):
        """测试递归深度递增"""
        result1 = self.agent.refract("困惑", depth=0)
        result2 = self.agent.refract("递归困惑", depth=1)
        self.assertEqual(result2["metadata"]["recursion_depth"], 1)
```

## 伦理承诺

### 核心伦理原则

#### 1. 非评判性原则
- 不暗示某种视角"更正确"
- 不贬低用户的现有理解
- 不强加特定的价值观

#### 2. 自主性保障
- 必须预留用户思考空间
- 用户决定是否深入探索
- 随时安全退出的权利

#### 3. 透明度要求
- 明确AI参与的身份
- 说明能力范围和局限
- 不伪装或隐藏技术中介

#### 4. 安全底线
- 防止有害内容生成
- 保护用户隐私
- 避免认知过载

### 贡献者伦理承诺

所有贡献者必须同意以下承诺：

```text
我承诺，在参与棱镜协议项目时：

1. 尊重多元视角，不追求单一"正确答案"
2. 保障用户自主，不制造技术依赖
3. 保持透明诚实，不隐藏技术局限
4. 维护安全底线，不开发有害功能
5. 遵循开放原则，不设置不当限制

我理解，违反这些承诺可能导致贡献被拒绝或撤回。
```

## 沟通指南

### 讨论礼仪

#### GitHub Issues
- 使用清晰的标题
- 提供完整的信息
- 保持建设性态度
- 及时回复反馈

#### Pull Requests

- 回应所有评论
- 保持代码整洁

#### 社区讨论
- 尊重不同意见
- 提供证据支持
- 避免人身攻击
- 保持专业态度

### 冲突解决

#### 升级路径
1. **直接沟通**：尝试与对方直接解决
2. **维护者调解**：请求维护者帮助调解
3. **社区投票**：重大分歧进行社区投票
4. **治理委员会**：最终由治理委员会决定

#### 解决原则
- 对事不对人
- 基于事实和证据
- 尊重协议规范
- 考虑社区利益

### 决策过程

#### 日常决策
- **代码合并**：维护者批准即可
- **文档更新**：相关领域专家审核
- **bug修复**：验证后即可合并

#### 重要决策
- **协议变更**：需要RFC流程和社区投票
- **架构调整**：需要技术委员会审核
- **伦理问题**：需要伦理委员会审核

#### RFC流程
1. **草案阶段**：提交RFC草案
2. **讨论阶段**：社区讨论反馈
3. **修改阶段**：根据反馈修改
4. **投票阶段**：社区投票决定
5. **实施阶段**：批准后实施

## 常见问题

### Q: 我是新手，从哪里开始？
A: 建议从"新手友好任务"开始，如文档改进或简单bug修复。

### Q: 如何提出协议变更？
A: 需要走RFC流程，先在GitHub Discussions中提出想法，然后提交正式RFC。

### Q: 贡献需要签署CLA吗？
A: 不需要单独的CLA，但贡献即表示你同意在CC BY-NC 4.0许可证下发布贡献。

### Q: 如何成为维护者？
A: 通过持续高质量贡献获得社区信任，现有维护者提名，社区投票通过。

### Q: 商业使用相关问题？
A: 协议采用CC BY-NC 4.0，商业使用需要单独授权，请联系项目负责人。

### Q: 如何报告安全问题？
A: 通过GitHub的安全报告功能或发送邮件到安全团队。
